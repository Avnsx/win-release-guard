## Policy Feed, Trust, And JSON Hardening

Runtime clients fetch public JSON plus a detached signature. They do not authenticate to GitHub and do not trust hosting alone.

| Boundary | Implementation |
| --- | --- |
| Signed policy bytes | `verify_policy_signature()` over exact JSON bytes |
| Trusted public keys | `win11_release_guard/data/trusted_policy_keys.json` |
| Key rotation | `verify_not_after_utc` bounds retiring/retired keys |
| Schema validation | `validate_policy_document()` and `policy_from_dict()` |
| Strict JSON | Duplicate keys, invalid UTF-8, non-finite numbers, wrong object top-level shape, and oversized payloads are rejected |
| Runtime fallback | Live signed remote JSON -> verified fresh cache -> verified stale cache -> signed bundled fallback -> `CHECK_INCOMPLETE` |
| Strict production | Fresh live signed remote JSON is required for production-green results |

The signed bundled policy remains a last-known-good fallback and has its own legacy compatibility tests. Do not describe bundled fallback as a fresh `0.3.0` policy feed.

## GitHub Release Lane

`release.yml` publishes clean source archives, not policy trust roots. It validates `vX.Y.Z` tags against `pyproject.toml`, runs compile/tests/source checks, writes `dependency-freshness.json` as generated output, creates and validates `dist/win11_release_guard-source.zip`, scans for secrets, then creates a draft GitHub Release by default. The generated release body links `CHANGELOG.md`, the matching detailed `docs/releases/vX.Y.Z.md` notes, the Pages dashboard, the public feed, and the separate PyPI Trusted Publishing lane. Only `release.yml` requests `contents: write`.

Publishing a GitHub Release can trigger the separate `pypi-publish.yml` workflow through the release `published` event. The PyPI workflow still owns its own package gates, artifact handoff, environment approval, and OIDC publishing.

## Security Notes

| Rule | Reason |
| --- | --- |
| Keep runtime clients unauthenticated to GitHub. | Public Pages JSON plus detached signature is the trust path. |
| Keep private signing material in `WIN11_RELEASE_GUARD_POLICY_SIGNING_KEY_B64`. | Only public verification keys belong in source. |
| Keep WUA secondary. | WUA diagnostics explain offers/history; signed policy decides target. |
| Keep build-first local truth. | Display labels can be stale or misleading on real Windows systems. |
| Keep PyPI OIDC-only. | Avoids long-lived publishing credentials in repository automation. |
| Keep Pages static and local-only assets. | GitHub Pages compatibility and no-token operation. |
| Keep `/api/v1` compatible for at least 24 months. | Public consumers depend on stable aliases. |
