## Policy Feed, Trust, And JSON Hardening

Runtime clients fetch public JSON plus a detached signature. They do not authenticate to GitHub and do not trust hosting alone.

| Boundary | Implementation |
| --- | --- |
| Signed policy bytes | `verify_policy_signature()` over exact JSON bytes |
| Trusted public keys | `win11_release_guard/data/trusted_policy_keys.json` |
| Key rotation | `verify_not_after_utc` bounds retiring/retired keys |
| Schema validation | `validate_policy_document()` and `policy_from_dict()` |
| Strict JSON | Duplicate keys, invalid UTF-8, non-finite numbers, wrong object top-level shape, and oversized payloads are rejected |
| Runtime fallback | Live signed remote JSON -> verified fresh cache -> verified stale cache -> signed bundled fallback -> `CHECK_INCOMPLETE` |
| Strict production | Fresh live signed remote JSON is required for production-green results |

The signed bundled policy remains a last-known-good fallback and has its own legacy compatibility tests. Do not describe bundled fallback as a fresh `0.3.1` policy feed.

## Source Diagnostics

Source Diagnostics are generated from Release Health HTML, Atom feed enrichment, parser observations, and source freshness/drift checks.

| Component | Role |
| --- | --- |
| `_source_diagnostics()` | Builds structured event payloads. |
| `_source_diagnostic_rows()` | Converts events/warnings/notices into dashboard rows. |
| `_validate_source_diagnostics()` | Validates schema shape. |
| `publish-policy.yml` | Blocks publish when an event has `severity: error`. |
| `sync_source_diagnostics_issues.py` | Workflow-side issue sync and dry-run reports using the built-in Actions token only. |
| CLI source checks | Print source URLs, event summaries, warnings, and published URL checks. |

Notice and warning events remain visible diagnostics; only error events block publish. GitHub Issues use fixed severity labels (`internals: notices`, `internals: warning`, `internals: error`) plus an internal body marker for dedupe. Manual issues without the exact marker are not changed, stale managed issues are closed, and reappearing managed issues are reopened. Browser JavaScript renders only static issue metadata and never writes to GitHub.

## GitHub Release Lane

`release.yml` publishes clean source archives, not policy trust roots. It validates `vX.Y.Z` tags against `pyproject.toml`, runs compile/tests/source checks, writes `dependency-freshness.json` as generated output, creates and validates `dist/win11_release_guard-source.zip`, scans for secrets, then creates a draft GitHub Release by default. The generated release body links `CHANGELOG.md`, the matching detailed `docs/releases/vX.Y.Z.md` notes, the Pages dashboard, Pages Wiki/changelog routes, the public feed, and the separate PyPI Trusted Publishing lane. Only `release.yml` and `sync-wiki.yml` request `contents: write`: release publication uses it for the GitHub Release asset, and Wiki sync uses it only to mirror `wiki/*.md` Markdown to the same repository's GitHub internal Wiki.

Publishing a GitHub Release can trigger the separate `pypi-publish.yml` workflow through the release `published` event. The PyPI workflow still owns its own package gates, artifact handoff, environment approval, and OIDC publishing.

## Wiki And Documentation

| Path | Role |
| --- | --- |
| `CHANGELOG.md` | Compact root release summary. |
| `docs/releases/v0.3.1.md` | Detailed maintainer release notes. |
| `wiki/Release-v0.3.1.md` | Compact human wiki page. |
| `docs/maintainer-guide.md` | Maintainer guide. |
| `wiki/Home.md`, `wiki/_Sidebar.md` | Local GitHub Wiki source navigation and static Pages Wiki source. |

The repository `wiki/` folder is source/staging for the GitHub internal Wiki and source for the static Pages Wiki. `publish-policy.yml` renders it into Pages HTML under `site/wiki/`. The renderer escapes raw HTML, converts GitHub Wiki links, emits visible generator warnings for broken or missing Wiki inputs, and may add local-only inline SVG topic icons without changing the Markdown source. The separate `sync-wiki.yml` workflow can mirror `wiki/*.md` into the live GitHub internal Wiki with the built-in Actions token, or upload a dry-run Markdown artifact for manual fallback.

`CHANGELOG.md` remains manually maintained source of truth. Newer sections are added at the top, older version sections remain below, and the Pages generator renders the history to `site/wiki/changelog/` plus per-version pages for auditability and release history.

## Security Notes

| Rule | Reason |
| --- | --- |
| Keep runtime clients unauthenticated to GitHub. | Public Pages JSON plus detached signature is the trust path. |
| Keep private signing material in `WIN11_RELEASE_GUARD_POLICY_SIGNING_KEY_B64`. | Only public verification keys belong in source. |
| Keep WUA secondary. | WUA diagnostics explain offers/history; signed policy decides target. |
| Keep build-first local truth. | Display labels can be stale or misleading on real Windows systems. |
| Keep PyPI OIDC-only. | Avoids long-lived publishing credentials in repository automation. |
| Keep Pages static and local-only assets. | GitHub Pages compatibility and no-token operation. |
| Keep `/api/v1` compatible for at least 24 months. | Public consumers depend on stable aliases. |
