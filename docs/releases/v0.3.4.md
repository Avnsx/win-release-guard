# v0.3.4 Release Notes

Purpose: document the concrete `0.3.4` release state for maintainers, release managers, and future agents. Source truth for new work remains the current code, tests, workflows, `pyproject.toml`, README, docs, local wiki source, and `AGENTS.md`.

## Scope

| Item | Current state |
| --- | --- |
| Package/program version | `0.3.4` from `pyproject.toml` and `win11_release_guard.version.package_version()` |
| Package name | `win11_release_guard` |
| Runtime identity | `win11_release_guard/0.3.4` for runtime user-agent, generator identity, and WUA client application ID |
| Python support | `>=3.10` with classifiers for 3.10, 3.11, 3.12, 3.13, and 3.14 |
| API compatibility | Policy `schema_version` remains `1`; public `api_version` remains `v1` |
| Verdict model | Unchanged: signed public policy JSON plus detached signature is the compliance authority |

`0.3.4` is a hardening release on top of `0.3.3`. It corrects user-facing
source-evidence wording, hardens date and Atom-fallback parsing, makes clean
archive validation deterministic, and unifies the generated GitHub Pages wiki
visual scale with the dashboard. No signed verdict, `required_baseline_build`
selection, `/api/v1` contract, Source Diagnostics issue-sync rule, or the 14-day
baseline-update notice window changed.

## Baseline Notice Copy And Evidence Wording

The dashboard baseline-update notice summary is now assembled from complete,
source-aware sentences. It credits MSRC only when
`security_evidence_source == "msrc_cvrf"`, attributes validated Support article
evidence to Microsoft Support when `security_evidence_source == "support_article"`,
and uses neutral wording when evidence is unavailable/unknown/none. When checked
evidence does not classify the update as security, it uses clear non-alarmist
wording. The earlier behavior could assert "MSRC confirms" for Support-only
evidence; that is fixed. The summary no longer emits a `B.;` punctuation artifact
and never leaks raw status enums (such as `not_security`) into human-facing copy;
the structured `security_evidence_source` / `security_evidence_status` values
remain in machine JSON for triage.

## Hardened Date Parsing

Impossible or malformed ISO-shaped Release Health source dates (for example
`2026-02-30`) degrade to no active baseline notice instead of raising
`ValueError` and aborting policy, dashboard, or manifest generation. Non-zero-
padded calendar dates such as `2026-6-9` are accepted and normalized to
`2026-06-09`. Date-only Microsoft precision is preserved and no exact time of day
is invented.

## Balanced Atom KB-only Fallback

Atom remains discovery for Support article hrefs, not a `/help/<KB>` resolver.
After exact KB+build and build-only matching fail, the KB-only fallback may
attach a build-agnostic candidate only when its KB matches, it exposes a safe
canonical `https://support.microsoft.com` href, it is not Preview/Out-of-band for
a normal broad target, it is unambiguous after canonicalization, and no
same-release-family explicit candidate contradicts the row build. An explicit
candidate for a different build family no longer blocks an otherwise-safe
build-agnostic fallback, while wrong-build, unsafe-URL, Preview/OOB, and ambiguous
candidates are still rejected. Exact KB+build matching (including the KB5094126
multi-build case across `26200.8655` and `26100.8655`) is unchanged, and missing
Atom hrefs still do not synthesize `/help/<KB>` URLs.

## Deterministic Clean Archive Validation

`python tools/export_clean_archive.py --validate` runs its inner extracted-archive
pytest gate in a controlled environment: plugin autoload disabled
(`PYTEST_DISABLE_PLUGIN_AUTOLOAD=1`), inherited `PYTEST_ADDOPTS` removed, inherited
`PYTEST_PLUGINS` removed, and the recursion guard plus required Python runtime
variables preserved. A developer shell exporting `--cov=...` via `PYTEST_ADDOPTS`
or a stale `PYTEST_PLUGINS` can no longer change, fail, or hang validation; real
test failures and timeouts remain fatal. The project declares no required pytest
plugins, so coverage is unchanged, and the historical `docs/releases/v0.3.1.md` is
now a required archive entry alongside `v0.3.2`, `v0.3.3`, and this `v0.3.4` note.

## Unified Pages Wiki Visual Scale

The generated wiki/changelog pages now match the dashboard's reading size at
normal browser zoom with no CSS/browser zoom, `transform: scale`, or viewport
tricks. The shared wiki shell sets a responsive root `font-size`, widens the
content column so tables and code blocks use the available horizontal space
(prose stays unified at roughly the same width), wraps long commands instead of
clipping them behind a horizontal scrollbar, contains stray overflow without
breaking the sticky sidebar, and adds a half-transparent, hover-revealed,
dependency-free copy button to code blocks. The dashboard scale is unchanged and
Pages output stays static with no external JS, CSS, fonts, CDNs, tokens, or
runtime GitHub API calls.

## Packaging And PyPI

| Field | Value |
| --- | --- |
| PyPI project | `https://pypi.org/project/win11-release-guard/` |
| End-user install | `python -m pip install win11_release_guard` |
| Console script | `win11_release_guard = "win11_release_guard.__main__:main"` |
| Package data | `win11_release_guard = ["data/*.json", "data/*.sig"]` |
| Build artifacts | wheel and sdist in generated `dist/` |
| Distribution check | `python -m twine check dist/*` |
| Publish workflow | `.github/workflows/pypi-publish.yml` |
| Publish method | PyPI Trusted Publishing / GitHub OIDC, no PyPI API token |

If the PyPI project is absent before first publish, configure a Pending Trusted
Publisher first. Pending Trusted Publisher setup does not reserve the package
name; if the name is already controlled by another owner, stop and report instead
of publishing. Do not add PyPI API tokens, Twine passwords, usernames, or
credentialed repository URLs. The signed bundled production policy and detached
signature are not regenerated by this local version bump.

## Unchanged Verdict Boundaries

- Signed public policy JSON plus detached signature remains the compliance verdict authority.
- Local Windows labels, WUA, Panther/setup logs, DISM, Event Logs, and Source Diagnostics remain diagnostic evidence.
- 25H2 remains the existing-device broad-fleet target.
- 26H1 remains excluded for existing-device targeting.
- `/api/v1` policy, signature, and manifest aliases stay backward-compatible.
- Notice Source Diagnostic events remain dashboard-only and are not GitHub-Issue-syncable.
- The baseline-update notice is informational dashboard output only, visible for 14 days.
